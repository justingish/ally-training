Exercise 2: Fix Keyboard Navigation
Difficulty: Medium

This custom dropdown component isn't keyboard accessible. Modify it to support keyboard navigation and proper ARIA attributes: 

<div class="custom-dropdown">
  <div class="dropdown-toggle" onclick="toggleDropdown()">
    <span id="selected-option">Select an option</span>
    <span class="dropdown-icon">▼</span>
  </div>
  <div id="dropdown-menu" class="dropdown-menu" style="display: none;">
    <div class="dropdown-item" onclick="selectOption('Option 1')">Option 1</div>
    <div class="dropdown-item" onclick="selectOption('Option 2')">Option 2</div>
    <div class="dropdown-item" onclick="selectOption('Option 3')">Option 3</div>
  </div>
</div>

<script>
function toggleDropdown() {
  const menu = document.getElementById('dropdown-menu');
  menu.style.display = menu.style.display === 'none' ? 'block' : 'none';
}

function selectOption(option) {
  document.getElementById('selected-option').textContent = option;
  toggleDropdown();
}
</script>
  • Replace the main toggle element with a button
  • Use appropriate ARIA attributes: aria-expanded, aria-controls, and aria-haspopup
  • Make items focusable and keyboard-selectable
  • Consider role="listbox" for the dropdown with role="option" for items
  • Add keyboard event listeners for Enter, Space, Up/Down arrows, and Escape 
<div class="custom-dropdown">
  <button 
    id="dropdown-button"
    class="dropdown-toggle" 
    aria-haspopup="listbox" 
    aria-expanded="false"
    aria-controls="dropdown-menu">
    <span id="selected-option">Select an option</span>
    <span class="dropdown-icon" aria-hidden="true">▼</span>
  </button>
  <ul id="dropdown-menu" class="dropdown-menu" role="listbox" tabindex="-1" style="display: none;">
    <li class="dropdown-item" role="option" tabindex="0" id="option-1">Option 1</li>
    <li class="dropdown-item" role="option" tabindex="0" id="option-2">Option 2</li>
    <li class="dropdown-item" role="option" tabindex="0" id="option-3">Option 3</li>
  </ul>
</div>

<script>
document.addEventListener('DOMContentLoaded', () => {
  const button = document.getElementById('dropdown-button');
  const menu = document.getElementById('dropdown-menu');
  const options = Array.from(menu.querySelectorAll('[role="option"]'));
  let activeIndex = -1;

  // Toggle dropdown
  button.addEventListener('click', () => toggleDropdown());

  // Handle keyboard navigation for button
  button.addEventListener('keydown', (e) => {
    if (e.key === 'ArrowDown' || e.key === 'Enter' || e.key === ' ') {
      e.preventDefault();
      if (menu.style.display === 'none') {
        toggleDropdown();
        activeIndex = 0;
        options[activeIndex].focus();
      }
    }
  });

  // Handle keyboard navigation within dropdown
  menu.addEventListener('keydown', (e) => {
    switch (e.key) {
      case 'ArrowDown':
        e.preventDefault();
        activeIndex = Math.min(activeIndex + 1, options.length - 1);
        options[activeIndex].focus();
        break;
      case 'ArrowUp':
        e.preventDefault();
        activeIndex = Math.max(activeIndex - 1, 0);
        options[activeIndex].focus();
        break;
      case 'Enter':
      case ' ':
        e.preventDefault();
        selectOption(options[activeIndex].textContent);
        break;
      case 'Escape':
        e.preventDefault();
        toggleDropdown();
        button.focus();
        break;
    }
  });

  // Handle option selection
  options.forEach((option) => {
    option.addEventListener('click', () => {
      selectOption(option.textContent);
    });
  });

  function toggleDropdown() {
    const isExpanded = menu.style.display !== 'none';
    menu.style.display = isExpanded ? 'none' : 'block';
    button.setAttribute('aria-expanded', !isExpanded);
    if (!isExpanded) {
      activeIndex = 0;
      options[activeIndex].focus();
    }
  }

  function selectOption(optionText) {
    document.getElementById('selected-option').textContent = optionText;
    toggleDropdown();
    button.focus();
  }
});
</script>

Difficulty: Medium

This form has several accessibility issues. Fix them:

<div class="contact-form">
  <div class="form-title">Contact Us</div>
  
  <div class="input-group">
    Name
    <input type="text" placeholder="Enter your full name"/>
  </div>
  
  <div class="input-group">
    Email
    <input type="text" placeholder="Enter your email address"/>
  </div>
  
  <div class="input-group">
    <span>Choose a topic:</span>
    <div>
      <input type="radio" name="topic" value="support"/> Support
      <input type="radio" name="topic" value="sales"/> Sales

  </div>

  • Use a proper <form> element
  • Add a <fieldset> and <legend> for the radio button group
  • Associate labels with form controls using the for attribute
  • Add proper id attributes to all form controls
  • Replace div.submit-button with a proper <button type="submit">
  • Add required attributes and proper error handling with ARIA attributes
  • Use appropriate input types (e.g., type="email")

Accessibility Self-Assessment Checklist

Use this checklist to evaluate your current web application's accessibility:

1. Semantic HTML

  • Do you use appropriate heading levels (<h1> through <h6>) in a logical order?
  • Are <button> elements used for interactive controls instead of styled <div> elements?
  • Are navigation menus wrapped in <nav> elements?
  • Do you use <article>, <section>, <aside>, <header>, and <footer> elements where appropriate?
  • Are <table> elements used for tabular data, with proper <th>, <thead>, etc.?

2. Keyboard Navigation

  • Can all interactive elements be accessed via the Tab key?
  • Is there a visible focus indicator for all interactive elements?
  • Can all functionality be operated using only the keyboard?
  • Are custom components like dropdowns and modals properly handling keyboard events?
  • Is focus properly managed when modals open/close or content changes?

3. ARIA Attributes

  • Are aria-label or aria-labelledby used for elements that need accessible names?
  • Are dynamic content changes announced via aria-live regions?
  • Are proper landmark roles used for primary page sections?
  • Is aria-expanded used for toggling controls like accordions and dropdowns?
  • Are decorative images and icons marked with aria-hidden="true"?

4. Forms

  • Do all form controls have properly associated labels?
  • Are validation errors announced to screen readers?
  • Are required fields clearly indicated both visually and for screen readers?
  • Are form groups like radio buttons wrapped in <fieldset> with <legend>?
  • Do error messages provide clear instructions on how to fix issues?

5. Images and Media

  • Do all informative images have meaningful alt text?
  • Do decorative images have empty alt attributes (alt="")?
  • Do videos have captions and audio descriptions?
  • Do audio files have transcripts?
  • Can media players be operated with a keyboard?

6. Color and Contrast

  • Do text elements meet WCAG AA contrast requirements (4.5:1 for normal text, 3:1 for large text)?
  • Is color not the only way to convey information?
  • Are focus indicators visible with sufficient contrast?
  • Are links distinguishable from regular text?
  • Have you tested the interface in high contrast mode?

Actions

For any "No" answers:

  1. Prioritize fixes based on impact and frequency of usage
  2. Create tickets for accessibility improvements
  3. Plan for regular accessibility testing in your development process
  4. Consider automated tools like axe, Lighthouse, or WAVE
